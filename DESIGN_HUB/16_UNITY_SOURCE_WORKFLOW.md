# 16_UNITY_SOURCE_WORKFLOW：Unity 源码学习与改造流程

## 定位

Unity 源码项目不是最终项目本体，而是学习 Unity 工程结构、理解真实项目组织方式、验证从 Web Demo 迁移到引擎工程的中间层。

当前路线不建议在没有 Unity 经验时直接从 0 创建 Unity 项目。

更推荐：

```text
先用 Web Demo 把玩法讲清楚；
再用可运行 Unity 源码项目学习真实工程；
最后再决定是否进入自己的 Unity 独立原型。
```

---

## 阶段目标

```text
v1.x = Unity 源码学习 / 改造阶段
```

目标不是从 0 搭 Unity 项目，而是：

1. 找到可运行的 Unity 源码项目；
2. 让 AI 协助修复兼容性问题；
3. 学习项目结构、场景、Prefab、脚本、资源和 UI；
4. 小步修改，让参考项目逐步接近 Web Demo 已验证的方向；
5. 判断是否值得进入 `v2.x Unity 独立原型阶段`。

---

## 推荐工作区

Unity 源码学习 / 改造阶段默认使用：

```text
UNITY_SOURCE/
```

建议结构：

```text
UNITY_SOURCE/
├── README.md
├── SOURCE_AUDIT.md
├── COMPATIBILITY_LOG.md
├── STRUCTURE_NOTES.md
├── MODIFY_PLAN.md
├── OriginalSource/
├── ModifiedSource/
├── Docs/
├── Tests/
└── Temp/
```

说明：

```text
OriginalSource/ 用于保存原始源码快照或导入说明；
ModifiedSource/ 用于保存 AI 协助改造后的版本；
Docs/ 用于保存源码分析和学习笔记；
COMPATIBILITY_LOG.md 用于记录兼容性问题和修复过程；
MODIFY_PLAN.md 用于记录小步改造计划。
```

## 源码筛选标准

优先选择：

- 明确写了 Unity 版本；
- 有完整运行说明；
- 项目规模小到中等；
- 玩法与你的目标方向接近；
- 许可证清楚；
- 不依赖大量付费插件；
- 不依赖复杂后端；
- 2D 项目优先；
- 教程项目优先于大型完整项目；
- 最近几年仍有人运行、维护或讨论。

---

## 不建议选择的源码

避免选择：

- Unity 版本非常老；
- 依赖大量 Asset Store 插件；
- 许可证不清；
- 体量巨大；
- 无法运行；
- 缺失关键资源；
- 代码结构混乱；
- 多人在线 / 后端复杂项目；
- 强依赖平台 SDK；
- 只上传部分 Assets，缺少完整 ProjectSettings 或 Packages。

---

## 源码体检清单

导入或改造任何 Unity 源码前，必须先检查：

```text
1. Unity 版本；
2. 许可证；
3. 是否允许学习、修改、商用；
4. 是否依赖付费插件；
5. 是否依赖后端服务；
6. 是否缺失资源；
7. 是否有运行说明；
8. 是否有构建说明；
9. 是否能在本地打开；
10. 是否能进入主场景；
11. 是否能运行核心流程。
```

体检结果建议写入：

```text
UNITY_SOURCE/SOURCE_AUDIT.md
```

---

## 首次运行流程

推荐流程：

```text
1. 不急着改代码；
2. 先按项目指定 Unity 版本打开；
3. 记录首次打开报错；
4. 让 AI 分析报错类型；
5. 优先修复依赖、API 过时、包版本和资源缺失问题；
6. 能打开主场景后，再进入源码结构学习；
7. 能跑通核心流程后，再开始小步改造。
```

---

## AI 兼容性修复规则

AI 可以协助修复：

```text
API 过时；
包版本冲突；
脚本编译错误；
命名空间缺失；
输入系统差异；
场景引用缺失；
简单资源路径问题。
```

AI 不应擅自处理：

```text
大规模重构；
替换核心框架；
删除大量脚本；
升级 Unity 大版本；
替换付费插件；
修改许可证相关文件；
绕过版权或授权限制。
```

兼容性修复必须记录到：

```text
UNITY_SOURCE/COMPATIBILITY_LOG.md
```

---

## 源码结构学习

跑通后，AI 应帮助整理：

```text
场景结构；
Prefab 结构；
核心脚本；
输入系统；
UI 系统；
角色 / 敌人 / 物件逻辑；
数据配置方式；
资源组织方式；
游戏循环；
保存 / 读取方式，如有。
```

建议输出到：

```text
UNITY_SOURCE/STRUCTURE_NOTES.md
Docs/UnitySourceAnalysis/
```

---

## 小步改造规则

源码学习阶段的改造必须小步进行。

每轮只允许：

```text
修改一个 UI 流程；
调整一个配置表；
替换一个占位资源；
修改一个简单交互；
增加一个调试开关；
补充一段注释或文档；
修复一个明确 Bug。
```

禁止一轮内：

```text
重写核心架构；
重写战斗系统；
重写地图系统；
替换整个 UI 框架；
迁移到另一个 Unity 大版本；
直接把 Web Demo 全量搬进 Unity。
```

---

## 和 Web Demo 的关系

Web Demo 用于表达“我们要什么”。

Unity 源码项目用于学习“真实游戏工程怎么做”。

二者关系：

```text
Web Demo 验证玩法和界面方向；
Unity 源码学习真实工程结构；
小步改造让 Unity 源码接近 Web Demo 的核心体验；
不要直接复制 Web Demo 代码到 Unity。
```

---

## 进入 Unity 独立原型阶段的条件

只有满足以下条件，才建议进入 `v2.x Unity 独立原型阶段`：

- Web Demo 核心玩法已验证；
- 至少一个 Unity 源码项目已经跑通；
- 已完成源码结构分析；
- 已完成若干小步改造；
- 已理解 Unity 项目基本结构；
- 策划确认需要创建自己的 Unity 原型；
- 已明确最小可运行范围。
