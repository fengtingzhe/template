---

## 界面清晰度原则

- 玩家必须知道下一步要做什么；
- 主要按钮必须明显；
- 关键资源 / 状态必须可见；
- 反馈必须及时；
- 不要让 UI 遮挡核心玩法区域；
- Demo 阶段不追求完整界面数量，但必须展示核心路径；
- UI 不应过早复杂化。

---

## 核心交互流程

请用流程图形式填写核心交互。

模板：

```text
玩家看到目标
↓
玩家执行核心操作
↓
系统给出即时反馈
↓
状态 / 资源 / 进度发生变化
↓
玩家看到下一步目标
```

## HUD 原则

Demo 第一版 HUD 只保留必要信息。

| UI 元素 | 作用 | 是否需要 |
|---|---|---|
| 核心资源 / 进度 | 告诉玩家当前状态 | 待填写 |
| 当前目标 | 告诉玩家下一步做什么 | 待填写 |
| 核心交互提示 | 帮玩家完成操作 | 待填写 |
| 结算提示 | 展示阶段结果 | 待填写 |
| Console / Dev 按钮 | 开发调试入口 | 是 |

不建议过早显示：

- 大量功能按钮；
- 复杂任务列表；
- 多层菜单；
- 与 Demo 目标无关的系统入口。

---

## 通用 Console / Dev 入口

所有可运行 Demo 默认应包含 `Console` / `Dev` 按钮。

通用 Console 功能只包括：

```text
音乐开关；
音效开关；
暂停 / 继续；
重置场景 / 重置 Demo；
显示 FPS。
```

当开启 FPS 显示后，FPS 数值应显示在游戏画面的右下角。

项目特定调试功能不应默认加入 Console，除非任务卡明确要求。
