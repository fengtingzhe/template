# 00_MASTER_PROMPT：AI 总控规则

你正在协助开发一个游戏项目。

你的任务不是自由发挥重新设计游戏，而是在现有设计约束内，把项目逐步实现成可运行、可验证、可迭代的 Demo。

当前阶段，本项目不追求 AI 全自动完成复杂商业游戏，而是追求快速搭建可试玩、可展示、可验证的高表达力 Demo。

---

## 工作原则

1. 先理解项目，再修改代码。
2. 先保证可运行，再增强表现。
3. 先做小步迭代，不要一次性大规模重构。
4. 不确定时，不要擅自决定，写入 `DESIGN_HUB/10_OPEN_QUESTIONS.md`。
5. 所有修改必须可追踪、可验证、可回滚。
6. 不要删除已有设计，除非任务明确要求。
7. 不要自作主张改变核心玩法、世界观、商业模式、数值方向。
8. 不要新增大型依赖，除非任务明确允许。
9. 不要为了工程完整性牺牲 Demo 表达目标。
10. Codex 执行任务前必须先做任务归属判断。

---

## 任务归属判断规则

每次 Codex 阅读完任务提示词后，必须先判断本轮任务的执行归属。

Codex 必须在正式开发前输出：

```text
任务归属判断：
- 本轮由 Codex 自己开发 / 拆分给 DeepSeek / Codex + DeepSeek 协作
- 判断理由：
- Codex 负责：
- DeepSeek 负责：
- Codex 审核方式：
```

判断原则：

1. 核心架构、跨文件整合、状态管理、复杂 Bug 修复、最终合并，优先由 Codex 执行。
2. 普通 UI 组件、重复性代码、配置表、文案、简单工具函数，可以拆给 DeepSeek。
3. DeepSeek 不负责最终合并，不负责核心架构，不负责高风险重构。
4. DeepSeek 产物必须由 Codex 审核后才能合并。
5. 如果任务不可拆分，Codex 必须说明为什么由自己完成。
6. 如果任务需要拆给 DeepSeek，Codex 应在 `AI_TASKS/DEEPSEEK_TASKS.md` 中记录任务卡。

---

## Demo 开发优先级

```text
先让人看懂；
再让人玩懂；
再让人觉得有趣；
最后才考虑能不能扩展成完整项目。
```

开发优先级：

```text
1. 先展示核心体验；
2. 再展示主要界面；
3. 再展示关键反馈；
4. 再补充数值和内容；
5. 最后才考虑工程完整性。
```

---

## 每次任务开始前

你必须确认：

- 当前任务目标是什么；
- 本轮要改哪些文件；
- 本轮不应该改哪些文件；
- 验收标准是什么；
- 该任务服务哪个 Demo 目标；
- 任务由 Codex 自己开发、DeepSeek 开发，还是 Codex + DeepSeek 协作。

---

## 每次任务完成后

你必须输出：

1. 修改文件列表；
2. 实现内容；
3. 如何验证；
4. 是否有未解决问题；
5. 是否有需要策划决策的问题；
6. 是否已更新 `AI_TASKS/CHANGELOG.md`；
7. 是否涉及 DeepSeek 任务拆分；
8. 如涉及 DeepSeek，是否已更新 `AI_TASKS/DEEPSEEK_TASKS.md`。

---

## 不确定问题处理

如果你不确定某个设计、数值、体验或技术边界，不要擅自决定。

必须写入：

```text
DESIGN_HUB/10_OPEN_QUESTIONS.md
```

等待人类制作人确认。
