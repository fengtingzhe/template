# 09_CONFIG_FIRST_RULE：配置优先规则

## 定位

本文件用于防止 AI 把策划需要频繁调整的内容硬编码进代码。

核心原则：

```text
策划可能频繁调整的内容，不得硬编码。
```

---

## 应配置化的内容

以下内容应优先放入配置文件：

```text
初始资源；
关卡目标；
步数 / 时间；
敌人血量；
伤害；
奖励；
掉落概率；
生成概率；
难度曲线；
道具效果；
任务目标；
UI 文案；
关卡列表；
棋盘 / 地图 / 房间数据；
经济产出和消耗；
升级成本；
冷却时间；
阶段目标。
```

---

## Web Demo 配置路径

Web Demo 阶段优先使用：

```text
WEB_DEMO/Data/config/
```

推荐文件：

```text
WEB_DEMO/Data/config/game_config.json
WEB_DEMO/Data/config/levels.json
WEB_DEMO/Data/config/balance.json
WEB_DEMO/Data/config/ui_text.json
```

---

## 辅助工具与配置

如果配置经常需要修改，应考虑提供辅助工具。

例如：

```text
关卡配置频繁修改 → 关卡编辑器；
数值频繁修改 → 数值模拟器；
掉落表频繁修改 → 掉落表编辑器；
剧情文本频繁修改 → 文本表导入导出工具。
```

工具规划应写入：

```text
DESIGN_HUB/17_AUXILIARY_TOOLS_PLAN.md
```

---

## 禁止事项

- 不要把关键数值写死在 `game.js`；
- 不要把关卡数据写死在逻辑代码中；
- 不要把 UI 文案散落在多个脚本里；
- 不要让策划为了调数值必须改代码；
- 不要为了配置化过度设计复杂后台。

---

## 例外情况

以下情况可以临时硬编码：

```text
v0.1 阶段用于验证最小核心操作的临时代码；
一次性测试参数；
明确不会被策划调整的内部常量。
```

但进入 `v0.3 配置化与基础数值` 之前，应清理关键硬编码。
