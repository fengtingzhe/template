# 03_TECHNICAL_RULES：技术与工程规则

本文件用于定义当前工作流的技术阶段、目录约定和工程边界。

当前模板不默认进入任何游戏引擎从 0 开发。

默认路线是：

```text
v0.x = Web Demo 原型阶段
v1.x = Unity 源码学习 / 改造阶段
v2.x = Unity 独立原型阶段
v3.x = 可展示垂直切片阶段
v4.x = 商业化验证阶段
```

---

## v0.x：Web Demo 原型阶段

默认工作区：

```text
WEB_DEMO/
```

默认技术：

```text
HTML + CSS + 原生 JavaScript
Vite 或等价本地 Web 服务
```

原则：

- 优先验证核心玩法和界面信息；
- 不追求最终工程架构；
- 不引入复杂前端框架，除非任务明确允许；
- 不把 Web Demo 源码放入 `Builds/`；
- 不把 Web Demo 配置、工具、测试散落到根目录 `Data/`、`Tools/`、`Tests/`。

---

## v1.x：Unity 源码学习 / 改造阶段

默认工作区：

```text
UNITY_SOURCE/
```

目标：

- 筛选可运行的 Unity 源码项目；
- 检查 Unity 版本、许可证、依赖插件和运行说明；
- 让 AI 协助修复兼容性问题；
- 学习真实 Unity 工程结构；
- 小步改造接近 Web Demo 已验证的方向。

原则：

- 不从 0 创建 Unity 项目；
- 优先跑通已有源码；
- 不急于改玩法，先确保项目能运行；
- 不擅自升级 Unity 版本；
- 不擅自替换大量资源；
- 不擅自删除原始源码结构；
- 所有兼容性修复必须记录。

---

## v2.x：Unity 独立原型阶段

默认工作区：

```text
UNITY_PROJECT/
```

进入条件：

- Web Demo 核心玩法已验证；
- Unity 源码学习阶段已理解基本工程结构；
- 策划确认有必要创建自己的 Unity 原型；
- 已明确目标平台、核心系统和最小可运行范围。

---

## 通用代码原则

- 优先保持简单；
- 不要过早抽象；
- 不要一次性大重构；
- 新增功能要尽量局部化；
- 状态管理必须清晰；
- 数据优先来自当前阶段对应工作区；
- 每次修改都必须说明验证方法；
- 不确定时写入 `DESIGN_HUB/10_OPEN_QUESTIONS.md`。

---

## 不应过早实现

- 完整后端；
- 正式广告 SDK；
- 正式支付系统；
- 完整账号系统；
- 多人同步；
- 复杂编辑器工具；
- 商业化上线工程；
- 与当前验证目标无关的大型系统。
